# FiVaTech (Fixed/Variant Tree)

FiVaTech (Kayed & Chang, IEEE TKDE 2010) merges **all** input DOM trees simultaneously into a single **fixed/variant pattern tree**, then classifies each node position as fixed (template) or variant (data). From this pattern tree the data schema and per-page values can be deduced automatically.

## Paper Background

The original paper frames web data extraction as a **page-level** problem: given a set of DOM trees generated by the same server-side template, deduce both the template and the data schema without any human annotation.

### Data Schema Types

The paper defines a hierarchical type system for the data embedded in pages (Definition 2.1):

| Type | Notation | Meaning |
|------|----------|---------|
| **Basic** | beta | A string of tokens -- the atomic data unit |
| **Tuple** | &lt;phi_1, ..., phi_n&gt; | An ordered list of sub-types (cardinality = 1) |
| **Optional** | (phi)? | Present or absent (cardinality 0 or 1) |
| **Set** | `{phi}` | A repeating collection (cardinality >= 1) |
| **k-tuple** | &lt;phi_1, ..., phi_k&gt; | A disjunction where exactly one sub-type is instantiated |

A real page often combines these: for example, a product listing page might have a tuple of (title, price, description) repeated in a set, with an optional discount field.

### Tree Templates and Encoding

Rather than treating pages as flat token strings (as EXALG does), FiVaTech models both the template **T** and the data schema as trees. The **encoding function** `lambda(T, x)` describes how a data instance `x` is plugged into the template to produce a DOM tree. For a type constructor of order *n*, the template has a parent template, *n+1* child templates, and *n* insertion positions specifying where data subtrees attach.

### Problem Formulation

Given *n* DOM trees `DOM_i = lambda(T, x_i)` all generated from the same unknown template T, the goal is to deduce T and the values `{x_1, ..., x_n}` from the DOM trees alone (Definition 2.3).

## Algorithm Overview

The paper describes four main steps applied at each internal node of the pattern tree being built. Our implementation restructures these into a pipeline that is easier to follow, but the core ideas remain.

### Paper's Four Steps

The paper processes each internal node by collecting its first-level children across all input trees into a **peer matrix** M (one column per tree, rows for child subtrees), then:

1. **Peer node recognition** (Section 3.1) -- Compare subtrees with the same root tag using tree edit distance. Subtrees whose normalized matching score exceeds a threshold delta are assigned the same symbol. Leaf text nodes and `<img>` elements use value equality instead.

2. **Peer matrix alignment** (Section 3.2) -- Align the rows of M so that each row contains either the same symbol across all columns, or leaf nodes of different symbols (variant data). The `peerMatrixAlignment` algorithm processes rows top-to-bottom, shifting columns using `span` and `checkSpan` scoring rules (R1, R2, R3) to handle misaligned children caused by set-type or optional data.

3. **Repetitive pattern mining** (Section 3.3) -- Scan the aligned childList for tandem repeats (consecutive runs of the same pattern). Each detected repeat is merged: all occurrences except the first are deleted, and the pattern is marked as **set-typed**.

4. **Optional node merging** (Section 3.4) -- Nodes that disappear in some columns of the aligned matrix are detected via **occurrence vectors**. Adjacent optional nodes with the same or complementary occurrence vectors are grouped under a single virtual node in the pattern tree.

After these four steps, the system recurses into each non-leaf child node, building the pattern tree top-down.

### Schema Deduction (Section 4)

Once the full fixed/variant pattern tree is constructed, the paper deduces the data schema by:
- Removing internal nodes with a single child (collapsing unnecessary nesting)
- Preserving all basic-type leaf nodes (the variant leaf positions)
- Marking node order using k-order tuples
- Identifying templates by segmenting a pre-order traversal at each reference node (tuple-type nodes)

## Our Implementation

Our implementation preserves the paper's core insight -- simultaneous N-way tree merge with fixed/variant classification -- but uses a different mechanism for child alignment. The paper's peer matrix alignment with span/checkSpan scoring is replaced by **LCS-based child sequence alignment**, which is simpler to implement and naturally extends to N pages.

### Phase 1: Structure Check

Verify that all pages share the same **root tag** and **attribute names**. This is the structural compatibility gate -- if root structures differ, the pages cannot have come from the same template.

:::info Paper connection
This corresponds to the paper's assumption that "all input DOM trees have the same root node" and the peer node recognition step's requirement that compared subtrees share the same root tag.
:::

### Phase 2: Child Sequences

Extract the **child tag sequence** from each page. For example, if page 1 has children `[h1, p, p]` and page 2 has `[h1, p, p, p]`, these sequences capture the structural variation.

### Phase 3: Backbone Computation

Compute the **Longest Common Subsequence** (LCS) of all child tag sequences progressively (fold pairwise). The result is the **backbone** -- the maximal common structural skeleton shared by all pages.

:::info Paper connection
The backbone serves the same role as the paper's aligned peer matrix rows that contain the same symbol across all columns. Where the paper uses peer matrix alignment with shifting rules (R1, R2, R3) to discover this common structure, we use LCS which is a well-known polynomial-time algorithm for the same underlying problem.
:::

#### Structural Keys

For alignment, we use **structural keys** rather than bare tag names: each child's key is its `tag + sorted attribute names`. This distinguishes elements like `<p>` from `<p class="highlight">` that are structurally different even though they share a tag. Child tags are deliberately excluded from the key because loop-bearing children have variable child counts across pages.

:::info Paper connection
This is our analog of the paper's peer node recognition (Section 3.1). The paper uses tree edit distance with a threshold delta to decide if two subtrees are "peers." Our structural key approach is a simpler approximation: two elements are peers if and only if they have the same tag and the same set of attribute names.
:::

#### Value-Aware Backbone Fallback

When no stable structural keys exist (all children have variable counts), the algorithm checks whether a single structural key is present in every page with at least one instance that has **identical values** across all pages while other instances differ. If so, that first instance is promoted to the backbone -- it is a fixed element cohabiting with loop items of the same tag.

### Phase 4: Alignment

Map each page's children to backbone positions using **greedy left-to-right matching**. Children that don't align to any backbone position fall into **gaps** between backbone positions.

### Phase 5: Gap Analysis

Classify each gap between backbone positions:

| Pattern | Classification | Paper equivalent |
|---------|---------------|-----------------|
| Same-tag children, max count > 1 | **Repeating region** (loop/set) | Set-type from repetitive pattern mining (Section 3.3) |
| Same-tag children, present in some pages only | **Optional region** | Optional nodes from occurrence vectors (Section 3.4) |
| Multi-tag gap, consistent tag order | **Per-tag decomposition** into repeating/optional | Mixed regions handled by the paper's R3 divide-and-conquer rule |

:::info Paper connection
The paper's repetitive pattern mining detects tandem repeats in the aligned childList and marks them as set-typed. Our gap analysis achieves the same goal: children in a gap with variable count are classified as repeating (set-type). The paper's optional node merging uses occurrence vectors (per-column presence/absence) to group optional nodes; our optional region detection uses the same per-page presence information.
:::

### Phase 6: Recursive Merge

For each backbone position, recursively merge the aligned children from all pages. At each recursive level:
- **Text** positions: fixed if all pages have the same value, variable otherwise
- **Tail** positions: same rule
- **Attribute values**: fixed if unanimous, variable otherwise
- **Children**: repeat the full alignment process recursively

Repeating regions merge all instances from all pages into a single loop body template. Optional regions merge only the pages where the region is present.

### Phase 7: Result

The final pattern tree with fixed/variable slots at every position, plus any detected repeating and optional regions.

## Key Differences from the Paper

| Aspect | Paper (Kayed & Chang 2010) | Our implementation |
|--------|---------------------------|-------------------|
| **Child alignment** | Peer matrix alignment with span/checkSpan shifting rules | LCS-based progressive backbone computation |
| **Peer node recognition** | Tree edit distance with threshold delta | Structural key equality (tag + attribute names) |
| **Repeat detection** | Tandem repeat mining on aligned childList | Gap analysis: variable-count same-tag children |
| **Optional detection** | Occurrence vectors with grouping rules | Gap analysis: same-tag present in subset of pages |
| **Processing order** | Depth-first top-down with peer matrix at each level | Same depth-first recursion, but alignment via `child_alignment.py` |
| **Schema deduction** | Full type hierarchy with tuples, sets, optionals, k-tuples | Flat variable map with repeating/optional region annotations |
| **N-way merge** | Simultaneous via peer matrix columns | Simultaneous via LCS of all sequences + recursive merge |

## Key Properties

- **Simultaneous merge**: examines all pages at once, unlike anti-unification's pairwise fold
- **Order-independent**: the result does not depend on the order pages are provided
- **LCS alignment**: handles structural variation (different child counts) gracefully
- **Structural keys**: distinguishes elements by tag + attribute names, not just tag
- **Nested region support**: handles repeating and optional regions at any depth
