# RoadRunner (ACME/UFRE)

RoadRunner (Crescenzi, Mecca & Merialdo, VLDB 2001) linearizes DOM trees into token streams and learns a **Union-Free Regular Expression** (UFRE) grammar via the ACME algorithm (Align, Collapse under Mismatch, Extract). The algorithm discovers template structure by examining *mismatches* between pages -- places where two pages from the same site differ reveal variable data, optional regions, and repeating patterns.

## Theoretical Background

### Nested Types and the Data Model

The paper models data-intensive web sites as scripts that serialize a **source dataset** of *nested type* into HTML. A nested type is built from string fields, ordered tuples, and possibly-empty lists -- allowing arbitrarily deep nesting. The template inference problem becomes: given HTML pages generated by the same script, recover the nested type and decode the original data.

### Union-Free Regular Expressions

A **Union-Free Regular Expression** (UFRE) over an alphabet of HTML tags and the special symbol `#PCDATA` (representing variable text) is defined using only:

- **Concatenation** (`a . b`) -- sequential tokens
- **Iterator** (`(a)+`) -- one or more repetitions (maps to list types)
- **Optional** (`(a)?`) -- zero or one occurrence (maps to nullable fields)

Crucially, UFREs have **no union operator** -- the grammar cannot express "either A or B." This restriction is what makes the class learnable from positive examples alone: a UFRE lattice exists with a well-defined least upper bound (LUB) operation. We can write `(a)*` as shorthand for `((a)+)?` (zero or more).

### The UFRE Lattice

Given a containment relation where `sigma_1 <= sigma_2` iff `L(sigma_1) is a subset of L(sigma_2)`, UFREs form a lattice. The LUB of two UFREs is the minimal UFRE whose language contains both inputs. Because LUB is associative, inferring a grammar from *k* pages reduces to iteratively computing `LUB(sigma_1, sigma_2)` for pairs. This is exactly what ACME does.

## The ACME Algorithm

ACME (**A**lign, **C**ollapse under **M**ismatch, **E**xtract) is the core matching procedure. It takes two inputs:

- A **wrapper** -- the current UFRE grammar (initially page 1's token stream as all-Literal)
- A **sample** -- a new page's token stream

It walks both streams in lockstep, parsing the sample against the wrapper. When they agree, it advances. When they disagree, it has discovered a **mismatch** that reveals something about the template structure.

### 1. Linearization

The first page's DOM tree is flattened into a token stream: open tags, attributes, text content, child tokens, tail text, close tags -- in document order. Each token records its kind (`open`, `close`, `text`, `tail`, `attr`), its value, and its canonical position key.

### 2. UFRE Initialization

The token stream is wrapped as a list of **Literal** elements -- each token must match exactly. This is the most specific possible UFRE for one page.

### 3. Mismatch-Driven Generalization

For each subsequent page, ACME compares the current wrapper against the new page's token stream. When tokens match, it advances. When they disagree, it classifies the mismatch and generalizes:

#### String Mismatches: Discovering Fields

When the wrapper and sample have the same structural token (same kind + tag + attribute name) but **different values**, this is a *string mismatch*. The differing position must be a database field. ACME generalizes by replacing the Literal with a **Var** (the paper's `#PCDATA`). For example, if page 1 has `"John Smith"` and page 2 has `"Paul Jones"` at the same position, that position becomes a variable.

Constant strings that happen to be the same across all training pages remain as Literals -- they are treated as fixed template content.

#### Tag Mismatches: Discovering Structure

When the wrapper and sample have **different structural tokens** at the same position, this is a *tag mismatch*. Tag mismatches reveal structural variation: regions that repeat a different number of times across pages, or regions present in some pages but not others.

The paper's strategy is to **try iterator first**, then fall back to optional. This ordering matters: if you find an optional where an iterator actually exists, subsequent pages will fail to match.

### 4. Iterator Discovery

When a tag mismatch might be caused by different-length repetitions (e.g., page 1 lists 2 books, page 2 lists 3), ACME uses a three-step procedure:

**Step 1 -- Terminal-tag search.** Since we are hypothesizing an iterator `(s)+`, both the wrapper and sample must contain at least one occurrence of the repeating unit *s*. The last token of *s* (the *terminal tag*) is the token immediately before the mismatch point. ACME searches forward through the wrapper and sample for additional occurrences of this terminal tag. Each occurrence defines a *candidate square* -- a potential repeating unit.

**Step 2 -- Backward square matching.** For each candidate, ACME verifies the hypothesis by matching the candidate occurrence **backwards** against the already-matched portion. Token by token, working from the terminal tag backward, it checks structural compatibility. If the backward match succeeds for the full candidate length, the square is verified.

**Step 3 -- Wrapper generalization.** Once verified, ACME searches for all contiguous occurrences of the square around the mismatch region and replaces them with `(s)+` (a Repeat element). The parsing then resumes after the iterator.

In the interactive trace, iterator resolution appears during the "ACME Fold" phase -- the trace shows **Repeat** elements (blue) in the resulting UFRE.

### 5. Optional Discovery

If no iterator candidate succeeds, ACME hypothesizes an **optional** -- content present on one side but not the other.

**Cross-search resynchronization.** Given mismatching tokens `w` (on the wrapper) and `s` (on the sample), ACME performs a *cross-search*: look for `w` in the sample's upcoming tokens, and look for `s` in the wrapper's upcoming elements. Whichever search succeeds first identifies which side has the extra (optional) content and where to resume. The skipped tokens are wrapped in an Optional block `(...)? `.

For example, if page 1 has no image and page 2 has `<img src="..."/>`, the image tokens appear only in the sample. Cross-search finds the wrapper's next token in the sample after the image, so the image region becomes optional.

### 6. Wrapper Generalization After Mismatch

After resolving any mismatch (string, iterator, or optional), ACME updates the wrapper to incorporate the newly discovered pattern and **resumes parsing** from the resynchronization point. The algorithm succeeds when all mismatches have been resolved and both streams are exhausted.

### 7. Result

The final UFRE grammar contains four element types:
- **Literal** (green): fixed tokens that must match exactly
- **Var** (orange): positions where values vary across pages (`#PCDATA` in the paper)
- **Optional** (purple): structural regions present in some pages but not others (`(?)`  in the paper)
- **Repeat** (blue): structural regions that repeat variable times (`()+` in the paper)

## Advanced Topics

### Recursion: Internal Mismatches

Mismatch resolution is inherently **recursive**. When ACME identifies a candidate square for an iterator, matching the square body against the wrapper may trigger *new* internal mismatches. For example, discovering a list of books may trigger an inner mismatch for each book's list of editions. The algorithm applies the same mismatch-resolution logic recursively to these internal mismatches.

### The AND-OR Tree and Backtracking

The full space of possible mismatch resolutions forms an **AND-OR tree** (the paper's Section 4.3). Each external mismatch is an AND node (all must be solved), and each mismatch's candidate resolutions are OR children (try alternatives). ACME explores this tree, backtracking when a chosen candidate leads to a dead end.

Our implementation bounds the search: at most `_MAX_CANDIDATES = 6` candidate squares per hypothesis, and at most `_MAX_LOOK = 50` tokens of lookahead for both iterator terminal-tag search and optional cross-search.

### The Fixpoint Heuristic

The paper observes that external-level iterator discoveries are very likely correct and proposes treating them as **fixpoints** -- never backtracking past a confirmed iterator. Our implementation follows this: once an iterator is committed at the top level, it stays. Only candidate selection within a single mismatch involves backtracking.

## Implementation Details

### Token Representation

Each DOM node is linearized into a sequence of tokens. A `Token` records five fields: `kind` (open, close, text, tail, attr), `tag`, `attr_name`, `value`, and `position_key`. The `structural_key` -- the tuple `(kind, tag, attr_name)` -- determines whether two tokens match structurally, while `value` carries the extractable content.

### Pairwise Fold Across N Pages

Like the paper, our implementation processes pages sequentially: the first page initializes the wrapper, then each subsequent page is folded in via ACME. This means the result can depend on page ordering -- a consequence of the pairwise LUB approach. In practice, with 3+ training pages, the algorithm converges to the same grammar regardless of order for most real-world templates.

### Post-Fold Repeat Detection (`_detect_repeats`)

When one page has zero loop iterations and another has N iterations, ACME cannot discover an iterator during the fold (there is no "previous iteration" to compare against). Instead, it creates an Optional for the N items. After each fold, a **post-processing pass** converts these Optionals to Repeats by:

1. Detecting when an Optional's content structurally matches preceding elements (absorb into Repeat)
2. Detecting internal repetition within an Optional (split into Repeat of sub-unit)
3. Merging adjacent structurally-matching Optionals into a single Repeat

This post-processing runs after every ACME fold so that subsequent folds see the correct Repeat structure.

### Optional-to-Repeat Promotion

A related mechanism handles the case where a fold creates an Optional (from a 0-vs-N mismatch) and a *later* fold's sample has additional matching items. During ACME, if a tag mismatch occurs near a recent Optional whose body matches the current sample tokens, the Optional is promoted to a Repeat and all matching iterations are consumed.

## Key Properties

- **Token-stream approach**: operates on linearized DOM, not tree structure directly
- **Pairwise fold**: result can depend on page ordering
- **Iterator-first strategy**: at tag mismatches, tries iterator before optional (per the paper)
- **Bounded search**: limits lookahead and candidate count to avoid combinatorial explosion
- **Fixpoint pruning**: external iterator discoveries are never backtracked

## Known Limitations

- **Nested loops with variable-length squares**: when a repeating unit itself contains variable-length content, the square length is not fixed, making terminal-tag search ambiguous. The implementation uses bounded candidate search as a mitigation.
- **Same-tag disambiguation**: when multiple elements share the same tag (e.g., multiple `<div>` nesting levels), cross-search and terminal-tag search can find spurious matches. The implementation uses structural keys (kind + tag + attr_name) rather than tags alone, which helps but does not fully solve this.
- **No union/disjunction**: following the UFRE formalism, the algorithm cannot express "either structure A or structure B" -- it can only generalize to optionals and iterators.
- **Zero-iteration bootstrapping**: when the first page has zero iterations of a loop, the loop body cannot be discovered until the post-fold `_detect_repeats` pass or until a later page triggers Optional-to-Repeat promotion.
